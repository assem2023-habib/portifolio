Velocity-driven glitch text on hover

A per-character glitch animation where corruption intensity, slice count, and recovery easing all scale with how fast the mouse enters the element — instead of playing one fixed effect every time.

4 min read Updated Jun 12, 2024 Intermediate
CSS Vanilla JS Animation requestAnimationFrame
1The Problem

A fixed glitch effect feels robotic

Most glitch-on-hover effects play the exact same animation regardless of how the user interacted with the element. A slow, deliberate mouse movement triggers the same chaotic RGB-split-and-shatter sequence as a fast swipe across the screen — there's no relationship between the user's gesture and the visual response.

This breaks the illusion the effect is trying to sell. A glitch is supposed to feel like data corruption caused by interference — and interference should scale with the force of whatever caused it. A gentle hover producing a full violent shatter reads as disconnected from the interaction, not premium.

2The Solution

Map mouse velocity to a single intensity value

Track the mouse's velocity with exponential smoothing, normalize it into a 0–1 intensity value, and derive every other parameter — slice count, RGB offset, particle count, recovery duration, even which easing curve to use — from that single number.

This turns one physical measurement into a coordinated system: a slow approach produces a few large, soft slices and a quick fade-out. A fast swipe produces dozens of thin slices, visible RGB splitting, flying pixel fragments, and an elastic spring-back recovery.

◈

Key idea: don't hardcode separate "light" and "heavy" animation variants. Derive everything from one continuous variable so the transition between intensities is smooth, not a hard switch.

3Implementation

Three steps to build it

aTrack and smooth mouse velocity
JavaScript 
let smoothVX = 0;
let lastMX = 0, lastT = 0;

window.addEventListener('mousemove', e => {
  const now = performance.now();
  const dt = Math.max(now - lastT, 1);
  const rawVX = (e.clientX - lastMX) / dt * 16;
  smoothVX = lerp(smoothVX, rawVX, 0.40); // exponential smoothing
  lastMX = e.clientX; lastT = now;
});
bConvert speed into a 0–1 intensity value
JavaScript 
function getIntensity() {
  const speed = Math.sqrt(smoothVX ** 2 + smoothVY ** 2);
  // 1.5 = dead zone (ignores tiny jitter), 26.5 = speed that maps to full intensity
  return clamp((speed - 1.5) / 26.5, 0, 1);
}
cDerive every visual parameter from intensity
JavaScript 
const intensity = getIntensity();

const sliceCount   = Math.round(lerp(8,  50,  intensity));
const maxShift     = lerp(3,  42,  intensity);
const rgbOpacity   = lerp(0.08, 0.50, intensity);
const recoveryMs   = lerp(80, 320, intensity);
const easing       = intensity < 0.4 ? easeOutCubic
                : intensity < 0.75 ? easeOutQuint
                : elasticOut; // spring overshoot only at high speed
4Before / After

Fixed timing vs. velocity-derived timing

The structural difference is small in code, but the perceived quality difference is large — the second version makes the element feel physically responsive.

Before 
el.addEventListener('mouseenter', () => {
  // same animation every single time
  runGlitch({
    slices: 24,
    shift: 20,
    duration: 300
  });
});
Identical result for a slow drift or a fast flick — no gesture awareness.
After 
el.addEventListener('mouseenter', () => {
  const i = getIntensity();
  runGlitch({
    slices: lerp(8,50,i),
    shift: lerp(3,42,i),
    duration: lerp(80,320,i)
  });
});
Every parameter scales continuously with how the user actually moved.
