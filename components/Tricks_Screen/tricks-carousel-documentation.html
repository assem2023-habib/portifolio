كاروسيل Tricks
توثيق تقني شامل

شرح دقيق لكل طبقة بصرية وكل سطر منطق في القسم النهائي: من الحد المتحرك (conic-gradient) إلى نظام الجسيمات المرسوم بـ Canvas، مروراً بآلية الكاروسيل الخماسي الموضع، وحتى نظام البناء المتسلسل لمحتوى كل بطاقة.

البنية

كيف تُبنى البطاقة الواحدة؟

كل بطاقة هي تكديس من 5 طبقات مستقلة فوق بعضها، كل طبقة لها مسؤولية واحدة فقط. فهم هذا التكديس هو مفتاح فهم كل التأثير.

الترتيب الزمني للظهور مهم بقدر الترتيب المكاني: التوهج والحد يبدآن فوراً عند وصول البطاقة للمركز، الجسيمات تبدأ رسمها في نفس اللحظة، وأخيراً محتوى البطاقة (badge, title, code, desc, tags) يظهر بترتيب متتالٍ بفجوات زمنية محسوبة بدقة — سنشرح كل واحدة من هذه في القسم الخاص بها.

طبقة 1

الحد المتحرك (Animated Conic Border)

هذا التأثير لا يستخدم أي صورة أو SVG، بل تقنية CSS خام بالكامل تعتمد على دوران تدرّج مخروطي حول البطاقة بشكل متواصل.

Layer 1 Rotating Conic Gradient Border 7s linear infinite

المبدأ: نضع padding:2px على البطاقة الخارجية، ونملأ هذا الـ padding بتدرّج conic-gradient يدور حول نقطة المركز. الطبقة الداخلية (trick-card-inner) تغطي كل شيء إلا هذا الإطار الرقيق، فيظهر فقط كخط حدّ متوهج يدور.

كيف يدور فعلياً: نستخدم @property --border-angle لتعريف متغيّر CSS من نوع <angle> قابل للأنيميشن (هذا أساسي — متغيرات CSS العادية لا يمكن أن تتحرك بـ keyframes، لكن مع @property يصبح المتصفح يعرف أنها زاوية حقيقية ويقدر يتحرك بسلاسة بينها).

CSS — تعريف الزاوية القابلة للحركة
@property --border-angle {
  syntax: '<angle>';       /* يخبر المتصفح: هذا رقم زاوية حقيقي */
  inherits: false;
  initial-value: 0deg;
}

.trick-card {
  padding: 2px;              /* سمك الحد المتحرك */
  background: conic-gradient(
    from var(--border-angle),  /* الزاوية التي تتحرك */
    color-mix(in srgb, var(--card-accent) 90%, transparent) 0%,
    rgba(255,255,255,0.04) 18%,
    rgba(255,255,255,0.02) 35%,
    color-mix(in srgb, var(--card-accent) 70%, transparent) 55%,
    rgba(255,255,255,0.03) 72%,
    color-mix(in srgb, var(--card-accent) 90%, transparent) 100%
  );
}

.is-active .trick-card {
  animation: rotateBorder 7s linear infinite;
}
@keyframes rotateBorder {
  to { --border-angle: 360deg; }   /* دورة كاملة كل 7 ثوان */
}

لماذا 6 نقاط توقف (color stops) في التدرّج؟

لو استخدمنا نقطتين فقط (لون البداية ولون النهاية)، سيبدو الدوران كخط واحد متجانس يلف بلا تفاصيل. بتوزيع 6 نقاط — تتبادل بين لون الفئة بشفافيات مختلفة (90%، 70%) ولون أبيض شبه شفاف جداً (4%، 3%، 2%) — نحصل على نمط "نبضات ضوء" متفرقة حول الحد، فيبدو كأن الإطار "ينبض" أثناء دورانه بدل أن يكون حلقة ضوء ثابتة الكثافة.

لماذا 7 ثوانٍ بالضبط؟

هذه مدة مدروسة: أسرع من ذلك (2-3 ثوانٍ) يجعل الحركة تشتت الانتباه عن محتوى البطاقة نفسها. أبطأ من ذلك (15+ ثانية) يجعل العين لا تلاحظ أن هناك حركة فعلاً. 7 ثوانٍ هي النقطة التي يشعر فيها الدماغ بـ"حياة" في العنصر دون أن يسحب التركيز من النص.

◈

الحد المتحرك يعمل فقط على البطاقة النشطة (is-active) — البطاقات الجانبية لها حد ثابت تماماً بلا أنيميشن، وهذا أحد الأسباب التي تجعل العين تعرف فوراً أي بطاقة هي المركزية حتى قبل النظر إلى حجمها.

طبقة 2

نظام الجسيمات (Canvas Particle Engine)

كل بطاقة تحمل عنصر <canvas> خاصاً بها بنفس أبعادها بالضبط، يُستخدم لرسم 22 جسيمة متلألئة تطفو وتنبض ببطء — لكن فقط حين تكون البطاقة في المركز.

Layer 2 Per-Card Canvas Particle System 22 particles · RAF loop

لماذا Canvas وليس عناصر DOM منفصلة؟ لو رسمنا كل جسيمة كـ div مستقل، سنحتاج 22 عنصر × 6 بطاقات = 132 عنصر DOM إضافي، كل واحد يحتاج إعادة حساب layout عند كل تحريك. الـ Canvas يرسم كل الجسيمات في عملية رسم واحدة (fillRect/arc متتالية) بدون أي تكلفة على شجرة DOM.

تركيبة الجسيمة الواحدة

JavaScript — بنية كل جسيمة
const particles = Array.from({length: 22}, () => ({
  x: Math.random() * canvas.width,
  y: Math.random() * canvas.height,
  r: (Math.random() * 1.3 + 0.5) * dpr,     // نصف قطر 0.5-1.8px
  baseAlpha: Math.random() * 0.5 + 0.15,    // شفافية أساسية 15%-65%
  phase: Math.random() * Math.PI * 2,    // طور عشوائي للنبض
  speed: Math.random() * 0.4 + 0.15,         // سرعة النبض الفردية
  vy: -(Math.random() * 0.12 + 0.03) * dpr,  // سرعة الطفو للأعلى
  vx: (Math.random() - 0.5) * 0.05 * dpr      // انحراف أفقي خفيف جداً
}));

كيف تنبض كل جسيمة بشكل مستقل؟

كل جسيمة لها phase عشوائي (نقطة بداية مختلفة على منحنى الجيب) وspeed عشوائي (سرعة نبض مختلفة). هذا يعني أنه حتى لو كانت كل الجسيمات تستخدم نفس دالة Math.sin()، فإنها لا تتزامن أبداً — كل واحدة "تتنفس" بمعدلها الخاص، فيبدو المشهد كغبار نجمي حقيقي بدل وميض موحّد مزيّف.

tw = (sin(t × 0.02 × speed + phase) + 1) / 2 ← يحول الموجة من [-1,1] إلى [0,1] alpha = baseAlpha × (0.35 + 0.65 × tw) ← الجسيمة لا تختفي بالكامل أبداً (حد أدنى 35%)

الحركة الفعلية والتدوير اللانهائي

كل جسيمة تتحرك للأعلى ببطء (vy سالبة) وبانحراف أفقي طفيف جداً (vx). عندما تخرج من أعلى الـ canvas (y < -4 * dpr) تُعاد إلى الأسفل فوراً — وكذلك أفقياً عند الحواف — فتبدو الحركة كتدفّق لا ينتهي بدل دورة واضحة يلاحظها العين.

لماذا يتوقف نظام الجسيمات على البطاقات غير النشطة؟

كل بطاقة تملك دالتي start() وstop() مستقلتين. الكاروسيل يستدعي sys.stop() على أي بطاقة تخرج من موضع المركز فوراً، وsys.start() فقط على البطاقة التي تدخله. هذا يعني أنه في أي لحظة، يوجد نظام جسيمات واحد فقط نشط على الصفحة كاملة — مهم جداً للأداء لأن 6 أنظمة متزامنة كانت ستثقل المتصفح بلا أي فائدة بصرية إضافية (الجسيمات على البطاقات الجانبية لن تُرى بوضوح بسبب التعتيم أصلاً).

JavaScript — التحكم بدورة حياة نظام الجسيمات
return {
  start() { if (alive) return; alive = true; resize(); tick(); },
  stop()  { alive = false; cancelAnimationFrame(raf);
            ctx.clearRect(0,0,canvas.width,canvas.height); },
  resize
};

// عند تحديد موضع كل بطاقة في الكاروسيل:
if (offset === 0) { sys.resize(); sys.start(); }   // البطاقة المركزية فقط
else { sys.stop(); }                            // كل الباقي متوقف تماماً
طبقة 3

الخلفية الداخلية وتفاصيل العمق

طبقة trick-card-inner هي السطح الذي يحمل كل المحتوى المرئي، وتحتوي على 3 تأثيرات بصرية دقيقة جداً مكدّسة فوق بعضها.

Layer 3 trick-card-inner radial + linear + texture + corner glow

الخلفية الأساسية: مزيج من radial-gradient خافت بلون فئة البطاقة من الزاوية العلوية اليسرى، فوق linear-gradient قطري غامق من #272729 إلى #1c1c1e — هذا يعطي عمقاً خفيفاً جداً (الزاوية العلوية أفتح بصرياً قليلاً) بدل لون مسطّح واحد.

النمط القطري الخفي (texture)

CSS — ::before texture
.trick-card-inner::before {
  background-image: repeating-linear-gradient(
    135deg,
    rgba(255,255,255,0.015) 0px,    /* خط شبه شفاف جداً */
    rgba(255,255,255,0.015) 1px,
    transparent 1px,
    transparent 14px              /* فجوة 14px بين كل خط */
  );
  opacity: .5;
}

شفافية 1.5% فقط — هذا النمط لا يُلاحظ بنظرة مباشرة، لكنه يكسر "ملل" السطح المسطّح تماماً عند النظر من قرب، تماماً كما تفعل ورقة معدنية مصقولة بدقة بدل سطح بلاستيكي رخيص.

لمعة الزاوية (corner glow)

طبقة ::after منفصلة تضيف دائرة توهج صغيرة (90×90px) في الزاوية العلوية اليمنى فقط، بلون فئة البطاقة بشفافية 30%. هذا يحاكي انعكاس ضوء على سطح لامع — تفصيل صغير لكنه يضيف "طبقة" بصرية إضافية للعمق الكلي.

طبقة 0 (خلف كل شيء)

التوهج الخلفي (Glow Halo)

z: -1 carousel-card::before opacity controlled by --glow-strength

هذه الطبقة تمتد خارج حدود البطاقة (inset:-30px) وتُرسم خلفها (z-index:-1) — دائرة radial-gradient بلون فئة البطاقة تتلاشى تدريجياً للخارج. هذا ما يعطي إحساس "البروز" الحقيقي ضمن مساحة الكاروسيل، وليس مجرد بطاقة أكبر حجماً من البقية.

--glow-strength = 0.55 عند البطاقة المركزية فقط --glow-strength = 0 على كل البطاقات الأخرى (غير مرئي تماماً)

القيمة تنتقل بـ transition:opacity .5s ease فتظهر بسلاسة عند وصول البطاقة للمركز وتختفي بسلاسة عند مغادرتها، بالتزامن مع حركة الـ translateX نفسها.

لماذا rotateY بدل الاعتماد على scale فقط؟

الانعطاف الخفيف (rotateY بين -12 و +12 درجة) بالتزامن مع perspective:1500px على الـ viewport يعطي إحساساً بأن البطاقات الجانبية "تنحني بعيداً" عن المستخدم في فضاء ثلاثي الأبعاد حقيقي، بدل أن تبدو كصور مسطّحة مصغّرة فقط. هذا الفرق دقيق لكنه يضيف طبقة واقعية معتبرة للكاروسيل ككل.

حساب الـ offset الدائري (circular wraparound)

JavaScript — منطق الالتفاف الدائري
let offset = vi - current;
if (offset > M / 2)  offset -= M;   // لو البعد أكبر من النصف، التف من الجهة الأخرى
if (offset < -M / 2) offset += M;   // مثال: M=6, current=0, vi=5 → offset=5 لكن الأقرب فعلياً -1

هذا يضمن أن الكاروسيل "يلتف" بسلاسة من آخر بطاقة إلى الأولى دون أن يقطع كل البطاقات الوسطى بصرياً — البطاقة الأخيرة تبدو دائماً كأنها "إلى يسار" الأولى مباشرة بدل الانتقال الفجائي.

--chrome-fade → يتحكم بالهيكل (الخلفية، الحد، الظل) --txt-fade → يتحكم بالنص فقط (العنوان، الوصف، الكود، الـ tags)

لو استخدمنا opacity واحدة على البطاقة كلها، النص يصبح شبه شفاف بنفس درجة الخلفية فيختلط معها ويصعب قراءته. بفصل القيمتين، الهيكل (الخلفية والحد) يبقى واضحاً كجسم محدد المعالم حتى عند offset=1، بينما النص فقط يخف — هذا يعطي إحساساً بأن البطاقة "موجودة وواضحة لكن محتواها ثانوي الآن" بدل أن تبدو البطاقة كلها على وشك الاختفاء.

نظام الفلترة (Frontend/Backend/Mobile)

الفلترة لا تُخفي البطاقات بصرياً فقط، بل تعيد بناء كامل الكاروسيل حول مجموعة فرعية من البيانات.

JavaScript — applyFilter()
function applyFilter(filter) {
  visibleIdx = data
    .map((d, i) => ({d, i}))
    .filter(o => filter === 'all' || o.d.cat === filter)
    .map(o => o.i);                    // مصفوفة فهارس جديدة بالكامل

  dotsWrap.innerHTML = '';                // إعادة بناء النقاط من الصفر
  visibleIdx.forEach((_, vi) => { /* نقطة جديدة لكل بطاقة مرئية */ });

  current = 0;                            // إعادة البدء من أول بطاقة في المجموعة المفلترة
  hasEnteredOnce = false;
  layout();                              // إعادة حساب كل المواضع فوراً
  setTimeout(() => { /* تفعيل is-active + تشغيل الـ stagger من جديد */ }, 80);
}

المصفوفة visibleIdx هي المصدر الوحيد للحقيقة بخصوص "ما هو مرئي الآن" — دالة layout() تتحقق من كل بطاقة: إن لم تكن فهرستها موجودة في visibleIdx، تُخفى فوراً (opacity:0, pointer-events:none) بصرف النظر عن موضعها السابق. هذا يجعل التبديل بين الفلاتر يبدو كـ"كاروسيل جديد" فعلياً، مع احتفاظ كامل بكل آليات الحركة والتعتيم والـ stagger.

Stagger Reveal — ظهور المحتوى بترتيب

كل عنصر داخل البطاقة لا يظهر دفعة واحدة، بل بترتيب زمني محسوب بدقة عبر data-step وtransition-delay.

الترقيم الديناميكي حسب وجود الكود

JavaScript — إعادة ترقيم الخطوات حسب المحتوى
const descStep = d.code ? 3 : 2;   // لو فيه كود، الوصف يصير خطوة 3 بدل 2
const tagsStep = d.code ? 4 : 3;   // والـ tags تتأخر خطوة كاملة

هذا يضمن أن البطاقات التي لا تحتوي على code block (مثل "Avoid layout shift with will-change") لا تترك فجوة زمنية فارغة في التسلسل — الوصف يظهر مباشرة في الخطوة 2 بدل الانتظار لخطوة 3 بلا سبب.

آلية CSS الأساسية للتسلسل

CSS — b-item base state + delays
.b-item {
  opacity: 0;
  transform: translateY(14px);
  transition: opacity .42s cubic-bezier(.22,.9,.3,1),
              transform .42s cubic-bezier(.22,.9,.3,1);
}
.is-active .b-item { opacity: 1; transform: translateY(0); }

/* كل خطوة تنطلق متأخرة عن السابقة بفجوة ~110ms */
.is-active .b-item[data-step="0"] { transition-delay: .05s; }
.is-active .b-item[data-step="1"] { transition-delay: .16s; }
.is-active .b-item[data-step="2"] { transition-delay: .27s; }

كل عنصر ينتقل من opacity:0 مع إزاحة 14px للأسفل، إلى opacity:1 وموضعه الطبيعي — لكن بفجوة 110 ميلي ثانية تقريباً عن العنصر السابق. هذا الإيقاع المتتالي هو ما يعطي إحساس "البطاقة تُبنى أمام عينيك" بدل ظهورها كصورة جاهزة فجأة.

المشكلة التي حلّها class is-settled

بدون معالجة، كل مرة تعود فيها بطاقة سابقة إلى المركز سيُعاد تشغيل الـ stagger من الصفر — وهذا مرغوب فيه عند أول دخول للبطاقة لكنه يصبح مزعجاً ومتكرراً عند التنقل ذهاباً وعودة بسرعة. لذلك أضفنا is-settled: أي بطاقة غادرت المركز ولو لمرة واحدة تنتقل لحالة "مستقرة" حيث عناصرها تتحرك بـ transition بسيط 0.5s بلا أي تأخير متدرّج، فتظهر فوراً وبسلاسة دون إعادة عرض المسرحية كاملة في كل مرة.

ظهور القسم بالكامل عند التمرير

قبل أن يبدأ أي شيء داخل الكاروسيل، القسم بأكمله (الفلاتر + الكاروسيل) ينزلق من اليمين عند دخوله نطاق الرؤية.

JavaScript — IntersectionObserver
const obs = new IntersectionObserver((entries) => {
  entries.forEach(en => {
    if (en.isIntersecting) {
      revealBlock.classList.add('visible');   // يفعّل الانزلاق من اليمين
      obs.disconnect();                       // مرة واحدة فقط، لا يتكرر بالتمرير لأعلى/أسفل
    }
  });
}, {threshold: 0.25});                          // يبدأ عند ظهور 25% من القسم
.reveal-section { opacity: 0; transform: translateX(60px); } .reveal-section.visible { opacity: 1; transform: translateX(0); } transition: 0.7s cubic-bezier(.22,.9,.3,1)

بعد اكتمال هذا الانزلاق، تبدأ بعدها بـ120 ميلي ثانية مرحلة الـ stagger reveal للبطاقة المركزية الأولى — أي أن المستخدم يرى تسلسلين متتاليين من الحركة: أولاً القسم كله ينزلق من الجانب، ثم محتوى أول بطاقة يُبنى أمامه قطعة بقطعة.
